This section covers the configuration of the **Ansible Automation Platform (AAP)**. Before setting up the UI, it is important to understand the automation logic that connects the AI's request to the OpenShift cluster.

---

## 🧠 Understanding the Automation Logic

The provisioning process relies on two core files that work together to translate conversational requests into actual infrastructure.

### 0. The Workbench Mapping (`catalog/mapping.yml`)
Openshift AI has preconfigured workbenches available to users. The mapping.yml contains all of the image sku names and compute requirements.
> Think of this as the **"Menu"**. When the AI interviews the professor, it identifies a `catalog_id` (e.g., `pytorch_gpu`). This mapping file provides the technical details for that ID, including:
* The specific **Container Image** to pull.
* **Resource Limits** (CPU/Memory).
* **GPU requirements** (which triggers the high-performance hardware profile).

### 0. The Provisioning Playbook (`playbooks/deploy_workbench.yml`)
This is our Ansible playbook. One file for any workbench requested, the agent fills in values with our **mapping.yml** keys and information from the user.
* **Variable Lookup:** Matches the `catalog_id` against the mapping file.
* **Dynamic Hardware Selection:** Automatically switches the workbench to the `nvidia-gpu` profile if the selected image requires a GPU.
* **Scaling:** Uses a loop (`with_sequence`) to create unique ServiceAccounts, Storage (PVCs), and Workbench Notebooks for the exact number of students requested.
* **Git Integration:** Automatically clones the professor's provided GitHub URL into every student's workspace upon startup.

## 🛠️ AAP Configuration Steps

### 1. OpenShift Credentials

To ensure your demo doesn't break due to an expired session token, we will create a dedicated **ServiceAccount** with a long-lived (1-year) token, scoped only to your lab namespace.

#### **A. Create the Identity in OpenShift**
Run these commands in your terminal to create the account and grant it the necessary role for your workbench project. This will also output information needed in the next step.

```bash
# 1. Create the ServiceAccount in the 'aap' namespace
oc create serviceaccount aap-provisioner -n aap

# 2. Grant it 'admin' rights to your lab project 
# (Replace 'jupyter-labs' if you named your project something else)
oc adm policy add-role-to-user admin system:serviceaccount:aap:aap-provisioner -n jupyter-labs

# 3. Generate a token that lasts for 1 year (8760 hours)
oc create token aap-provisioner -n aap --duration=8760h
```

> 📋 **Copy the output:** The long string of text generated by the last command is your persistent credential. Step B will require this token and the URL generated by the following command.

```bash
# 4. Output the API Endpoint for your cluster
oc whoami --show-server
```
> 📋 **Copy the output:** This is the cluster API Endpoint. Feel free to start step B and come back here once you've pasted your credential.
---

#### **B. Configure the Credential in AAP**
Now, we need to hand this "key" to the Ansible Automation Platform.

1.  Log into the **AAP Web Console**.
2.  Navigate to **Automation Execution** > **Infrastructure** > **Credentials**.
3.  Click **Add**.
4.  Configure the following fields:

| Field | Value / Action |
| :--- | :--- |
| **Name** | `OpenShift SA Credential` |
| **Credential Type** | Select **OpenShift or Kubernetes API Bearer Token** |
| **API Endpoint** | Run `oc whoami --show-server` to find it |
| **Token** | Paste the **token** you generated in Step A |
| **Verify SSL** | **Uncheck** |

5.  Click **Save**.

---

### 2. Project & Inventory Configuration
The AI requires a synced codebase and a defined inventory to know where to execute the automation.

#### **A. Sync the Code**
1. Navigate to **Automation Execution** > **Infrastructure** > **Projects**.
2. Click **Add**.
3. **Name:** `aiops codebase`
4. **Source Control Type:** `Git`
5. **Source Control URL:** *Paste this Git repository URL*
6. Click **Save**.

#### **B. Create the Localhost Inventory**
Ansible needs a "target" even when running locally. We will create an inventory containing only `localhost`.
1. Navigate to **Automation Execution** > **Infrastructure** > **Inventories**.
2. Click **Add** > **Add inventory**.
3. **Name:** `Localhost`
4. Click **Save**.
5. Once saved, click the **Hosts** tab and click **Add** > **Add host**.
6. **Name:** `localhost`
7. **Variables:** Paste the following into the variables box:
   ```yaml
   ansible_connection: local
   ansible_python_interpreter: "{{ ansible_playbook_python }}"
   ```
8. Click **Save**.

---

### 3. Automation Template (Deploy_Workbench)
This template defines exactly how the AI triggers the provisioning playbook.

1. Navigate to **Automation Execution** > **Templates**.
2. Click **Create template** > **Create job template**.
3. Configure the template using the values in the table below:

| Configuration Field | Value to Enter / Select |
| :--- | :--- |
| **Name** | `Deploy Workbench` |
| **Job Type** | `Run` |
| **Inventory** | `Localhost` |
| **Project** | `aiops codebase` |
| **Playbook** | `playbooks/deploy_workbench.yml` |
| **Credentials** | `OpenShift Lab Credential` |
| **Extra variables** | 🛑 **CHECK: Prompt on Launch** |

> ⚠️ **MANDATORY:** You **MUST** check the **"Prompt on Launch"** box for **Variables**. If this is left unchecked, the AI will send the details, but AAP will ignore them, causing the deployment to fail.

4. Click **Save**.

---

### 4. Manual Approval Workflow (Human-in-the-Loop)
To prevent the AI from accidentally spinning up dozens of costly GPU instances without oversight, we wrap the template in a workflow:

1. Navigate to **Automation Execution** > **Templates**.
2. Select **+Create template** > **Create workflow job template**.
3. Give it a name , Check *Prompt on Launch* for Extra vairables, and press create.
> ⚠️ **MANDATORY:** You **MUST** check the **"Prompt on Launch"** box for **Variables**. If this is left unchecked, the AI will send the details, but AAP will ignore them, causing the deployment to fail.

4. Click **View workflow visualizer**
5. Add a step: **Node type - Approval**, give it a name, and press next + finsh.
6. Use the three dots on the Approval node and select **Add step and link**.
7. Select your `Deploy_Workbench` under **Job template**.
8. Change **Status** to `Run on success` and press next + next + finish.
9. **Save** your workflow
